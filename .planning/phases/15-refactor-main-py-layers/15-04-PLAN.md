## Decisions locked by this plan

**D-04-01 — Limiter access pattern: import from main.py.**
The `limiter` instance lives in `api/main.py` (it's tightly coupled to the FastAPI `app` instance via `app.state.limiter` and the rate-limit handler). Route modules import the limiter via `from api.main import limiter` to use `@limiter.limit(...)` decorators on route handlers.

Rejected alternative: `request.app.state.limiter` Depends pattern. Rejected because slowapi's `@limiter.limit(...)` is a *decorator*, not a runtime dependency — it must be evaluated at module import time. The decorator approach is the canonical slowapi pattern documented at https://slowapi.readthedocs.io and matches every existing handler in main.py without behavioural change.

Circular-import risk: `api/main.py` imports from `api/routes/*` AFTER it defines `limiter` (i.e., after the line `limiter = Limiter(...)` at main.py:162); route modules import `from api.main import limiter` at their own module top. This is a circular import in the strict sense, but Python tolerates it because by the time `api.main` reaches `app.include_router(...)`, the `limiter` symbol is already bound. The route-module imports happen at the BOTTOM of `api/main.py` (after `limiter = Limiter(...)`); route modules at the TOP do `from api.main import limiter` — Python only re-enters the partially-initialised `api.main` to *read* a symbol that already exists, not to run code.

**D-04-02 — `MAX_USERS` access in admin route: re-import from config + test patches both namespaces.**
The new `api/routes/admin.py` does `from api.config import MAX_USERS` at module top. The current test `test_admin_create_user_429_after_register_limit` (test_rate_limiting.py:236) patches `m.MAX_USERS` (i.e., `api.main.MAX_USERS`). After Plan 04 the route reads its own bound name `MAX_USERS` from `api.routes.admin` namespace; the existing patch on `api.main.MAX_USERS` no longer affects the route. Resolution: in Task 6 (admin route), the executor MUST also UPDATE the test to additionally patch `api.routes.admin.MAX_USERS`. This is a documented test fix, mirroring the Plan 03 pattern where 3 test files were updated to patch `api.services.auth_service.USERS_FILE` instead of `api.main.USERS_FILE`. SUMMARY notes the change.

**D-04-03 — `app.state.db` and `app.state.orchestrator` are set in lifespan startup.**
Lifespan startup adds:
```python
application.state.db = _db
application.state.orchestrator = get_orchestrator()
```
right after `await _db.startup(...)`. The deps providers `get_db(request)` returns `request.app.state.db`; `get_orchestrator_dep(request)` returns `request.app.state.orchestrator`. The deps function is named `get_orchestrator_dep` (not `get_orchestrator`) because `api.orchestrator.get_orchestrator` already exists as the singleton accessor — naming collision avoided.

**D-04-04 — Service signature change is breaking but additive in this plan.**
`auth_service._revoke_token` and `search_service._save_quota`, `_log_search`, `_stream_search` change signatures to add `db` (and `orch` for `_stream_search`) as explicit parameters. Module-level `from api.db import db as _db` is removed from auth_service.py and search_service.py at the same commit that adds the parameter. Internal helper-to-helper calls inside `_stream_search` thread `db` through to `_save_quota` and `_log_search`. The order is: Task 1 (deps providers) → Task 2 (service signature changes) → Task 3 (lifespan wiring + temporary inline call updates in main.py route handlers using local `db = _db` binding) → Tasks 4–8 (route extractions) → Task 9 (verification). **Critical: Task 3 keeps tests green by adapting the still-in-main.py route handlers to pass `db=_db` to the new service signatures, before Tasks 4–8 move the routes out.**

**D-04-05 — `api/main.py` post-plan keeps a small re-export shim for tests.**
After all routes move, `api/main.py` keeps these names at module level so existing test monkeypatches survive:
- `_db` (re-imported from `api.db`) — test_endpoints.py monkeypatches it
- `_create_token`, `_decode_token` — test_endpoints.py imports them
- `JWT_SECRET`, `JWT_ALGORITHM`, `MAX_USERS` — already imported via `from api.config import ...`, no change
- `limiter` — defined here, exported for routes (D-04-01) and tests
- `app` — the FastAPI instance, the test entry point

Anything else (constants only used by lifespan, internal helpers) can be trimmed if unused — Task 9's grep sweep handles this.

**D-04-06 — Commit granularity: 9 atomic commits, one per task.**
Each task ends with `pytest -q` green. If commit N regresses, revert N alone. Tasks 1–3 prepare contracts without moving routes (tests still green because routes still exist in main.py). Tasks 4–8 move one router at a time (tests still green because each move is small and `app.include_router(...)` is added in the same commit). Task 9 is the verification gate.

    ```python
            "version": "3.0.0",
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "memory_used_mb": round(mem_mb, 1),
            "memory_pct": mem.percent,
            "rss_mb": round(proc.memory_info().rss / 1024 / 1024, 1),
            "cpu_pct": cpu,
            "swap_used_mb": round(swap.used / 1024 / 1024, 1),
            "agents_paused": degradation != DegradationMode.NORMAL,
            "cache_entries": len(_api_cache),
            "uptime_s":             uptime_s,
            "active_tasks":         orch.active_count,
            "semaphore_slots_free": orch.semaphore_slots_free,
            "wal_size_bytes":       wal_size_bytes,
            "degradation_mode":     degradation.value,
        }


    @router.get("/health/memory")
    @limiter.limit(RL_ADMIN_LIMIT)
    async def health_memory(request: Request, _: dict = Depends(get_admin_user)):
        """Detailed memory profiling snapshot — admin only."""
        proc = psutil.Process()
        mem_info = proc.memory_info()
        mem = psutil.virtual_memory()
        traced_current, traced_peak = tracemalloc.get_traced_memory()

        snapshot = tracemalloc.take_snapshot()
        top_stats = snapshot.statistics("lineno")[:15]

        return {
            "rss_mb": round(mem_info.rss / 1024 / 1024, 1),
            "vms_mb": round(mem_info.vms / 1024 / 1024, 1),
            "system_memory_pct": mem.percent,
            "tracemalloc_current_mb": round(traced_current / 1024 / 1024, 2),
            "tracemalloc_peak_mb": round(traced_peak / 1024 / 1024, 2),
            "top_allocations": [
                {
                    "file": str(stat.traceback),
                    "size_kb": round(stat.size / 1024, 1),
                    "count": stat.count,
                }
                for stat in top_stats
            ],
            "cache_size": len(_api_cache),
            "cache_maxsize": _api_cache.maxsize,
            "agents_paused": get_orchestrator().degradation_mode != DegradationMode.NORMAL,
        }
    ```
