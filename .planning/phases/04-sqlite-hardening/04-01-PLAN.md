## Goal

Eliminate "database is locked" under concurrent agent load by replacing 9 scattered `aiosqlite.connect()` calls with a single persistent connection, WAL mode, and write serialization via `asyncio.Queue`.

---

## Current State (Problem)

**9 connection sites** in `api/main.py` — each opens, operates, and closes a fresh connection:

| Line | Function | Type |
|------|----------|------|
| 112 | `_save_quota()` | WRITE |
| 135 | `_init_rate_table()` | DDL+WRITE |
| 150 | `_check_rate()` | READ+WRITE |
| 306 | `_check_blacklist()` | READ+WRITE (purge) |
| 332 | `_revoke_token()` | WRITE |
| 375 | `_init_audit_db()` | DDL |
| 420 | `_log_search()` | WRITE |
| 1105 | `admin_stats()` | READ |
| 1175 | `admin_logs()` | READ |

**Why this breaks:** Each `aiosqlite.connect()` opens a new connection. Under concurrent searches (3+ SSE streams), multiple writers contend on the same database file. Without WAL, readers block writers and vice versa. On a 1vCPU VPS, this causes `OperationalError: database is locked`.

**Additional issue:** `quota_log` table is created inline at line 113 (inside `_save_quota`), meaning schema depends on which function runs first. This is fragile.

## Architecture

### New file: `api/db.py`

Single-responsibility module that owns the database lifecycle:

```
api/db.py
├── DatabaseManager (class)
│   ├── _conn: aiosqlite.Connection (single persistent connection)
│   ├── _write_queue: asyncio.Queue
│   ├── _writer_task: asyncio.Task (processes queue serially)
│   │
│   ├── startup() → init connection + PRAGMAs + DDL + start writer
│   ├── shutdown() → drain queue + close connection
│   │
│   ├── read(sql, params) → execute read query (direct, no queue)
│   ├── read_all(sql, params) → fetchall with Row factory
│   ├── read_one(sql, params) → fetchone with Row factory
│   │
│   ├── write(sql, params) → put on queue (fire-and-forget)
│   ├── write_await(sql, params) → put on queue + wait for result
│   │
│   └── execute_ddl(sql) → direct DDL (only during startup)
│
└── db: DatabaseManager (module-level singleton)
```

### Key design decisions

1. **Single connection** — SQLite is single-writer by design. One connection with WAL mode gives us concurrent reads + serialized writes. No "connection pool" (that's a PostgreSQL pattern, harmful for SQLite).

2. **WAL mode** — Readers don't block writers, writers don't block readers. Only one writer at a time (enforced by our queue, not by SQLite's lock).

3. **asyncio.Queue for writes** — All write operations go through a queue processed by a single background task. This eliminates lock contention entirely. Reads go direct (WAL allows concurrent reads).

4. **PRAGMAs at startup:**
   ```sql
   PRAGMA journal_mode=WAL;
   PRAGMA synchronous=NORMAL;    -- safe with WAL, 2x faster than FULL
   PRAGMA busy_timeout=5000;     -- wait 5s instead of failing immediately
   PRAGMA cache_size=-8000;      -- 8MB cache (default is 2MB)
   PRAGMA wal_autocheckpoint=100; -- checkpoint every 100 pages
   ```

5. **Schema consolidation** — All `CREATE TABLE IF NOT EXISTS` moves to `startup()`. No more inline DDL scattered across functions.

### Task 3: `_check_rate()` atomicity

The rate limiter needs atomicity: delete expired + count + insert must happen without interleaving. Solution:

```python
async def _check_rate(key: str, max_calls: int, window_s: int) -> bool:
    now = time.time()
    cutoff = now - window_s

    # Read current count (concurrent reads OK with WAL)
    row = await db.read_one(
        "SELECT COUNT(*) as cnt FROM rate_limits WHERE key = ? AND ts >= ?",
        (key, cutoff)
    )
    count = row["cnt"] if row else 0

    if count >= max_calls:
        # Purge expired entries in background
        await db.write("DELETE FROM rate_limits WHERE key = ? AND ts < ?", (key, cutoff))
        return False

    # Insert new entry + purge expired (both writes, serialized by queue)
    await db.write("DELETE FROM rate_limits WHERE key = ? AND ts < ?", (key, cutoff))
    await db.write("INSERT INTO rate_limits (key, ts) VALUES (?, ?)", (key, now))
    return True
```

This is safe because:
- Reads are non-blocking with WAL
- Writes are serialized through the queue
- Worst case: two concurrent checks both read count=9 (limit=10), both insert → count goes to 11 temporarily. Next check corrects. Acceptable for rate limiting (not financial transactions).
